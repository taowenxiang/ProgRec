# PRD: ProgRec 本科生科研导师匹配 Agent

## 摘要
- 项目定位：一个面向本科生的科研推荐助手，主任务是“找导师”，同时输出相关科研项目和潜在队友，形成从“想做科研”到“找到入口”的完整路径。
- 作业适配：严格按课程要求设计为 `1 个 Agent + 5 个独立 Skill`，每位组员负责 1 个 Skill，所有 Skill 可独立测试、可复用、可注册到 StudyClawHub。
- 产品风格：走“产品感增强型”路线，但保留课程项目所需的社会网络分析核心，强调可解释推荐、图谱关系和可量化评估。
- 目标用户：首次或早期尝试做科研的本科生，能够提供基础画像以及简历/经历文本，希望找到合适导师、切入项目和合作同学。
- 截止约束：PRD 默认服务于 `2026 年 5 月 15 日` 前完成的课程最终项目交付。

## 产品定义
- 核心问题：本科生缺少对导师、项目和同伴网络的可见性，不知道“我适合跟谁做、从哪里开始、和谁一起做”。
- 核心价值：把公开学术关系网络与校园场景数据结合，给出“导师主推荐 + 项目辅助推荐 + 队友辅助推荐”的一体化科研入口建议。
- 首版成功标准：用户输入兴趣方向、技能、年级、简历文本后，系统能够返回 Top-K 导师列表，并为每位导师附上相关项目、潜在队友和可视化解释。
- 非目标：首版不做完整聊天产品，不做跨校真实交易式撮合，不承诺真实联系导师或自动发邮件。

## Agent 设计
- Agent 名称：`ProgRec Agent`
- Agent 输入：`student_profile`、`resume_text`、`research_interest_query`、可选筛选条件（方向、导师风格、时间投入、技能要求）。
- Agent 输出：`mentor_recommendations`、`project_recommendations`、`teammate_recommendations`、`reason_graph`、`summary_report`。
- 编排流程：用户画像解析，异构科研网络构建，导师候选召回，项目与队友候选补充，联合排序与重排，解释生成与结果汇总。
- 决策逻辑：优先优化导师匹配分数；项目和队友不单独主导排序，而是作为“可进入该导师科研路径的证据与落地建议”增强结果可信度。
- SNA 核心：利用导师合作网络、研究社区划分、学生技能互补网络、项目关联图和图距离特征支撑推荐与解释。

## Skill 设计
1. `Student Profiling Skill`
- 负责人：组员 1
- 功能：解析学生输入与简历文本，抽取研究兴趣、技能标签、课程背景、项目经历、可投入程度。
- 输入接口：原始表单字段与简历文本。
- 输出接口：结构化学生画像 JSON 与文本/主题向量表示。
- 独立评估：标签抽取准确率、字段完整率、人工一致性检查。

2. `Academic Graph Builder Skill`
- 负责人：组员 2
- 功能：融合公开学术数据与模拟校园数据，构建异构图谱。
- 数据节点：导师、论文、研究主题、项目、学生。
- 数据边：合作、指导、主题相似、项目参与、技能互补、共同兴趣。
- 输出接口：标准化图数据对象与可查询索引。
- 独立评估：图构建覆盖率、实体对齐成功率、图连通性与统计摘要。

3. `Mentor Discovery Skill`
- 负责人：组员 3
- 功能：基于主题匹配、社区发现、网络邻近性和导师活跃度生成导师候选集。
- 方法重点：研究社区检测、导师影响力/中心性、主题相关性召回。
- 输出接口：导师候选列表及每个候选的图特征。
- 独立评估：Recall@K、候选覆盖率、社区划分可解释性。

4. `Project & Teammate Discovery Skill`
- 负责人：组员 4
- 功能：围绕已召回导师，为学生补充可切入项目和潜在队友，强调技能互补与进入门槛合理性。
- 方法重点：项目主题匹配、技能缺口分析、队友互补度、二跳社交路径。
- 输出接口：每位导师关联的项目建议与队友建议。
- 独立评估：匹配相关性人工评分、技能互补度得分、推荐多样性。

5. `Social Ranking Skill`
- 负责人：组员 5
- 功能：对导师、项目、队友进行联合打分与重排，形成最终结果。
- 方法重点：多目标排序，综合主题相似度、网络距离、社区归属、技能匹配、冷启动补偿与结果多样性。
- 输出接口：最终排序结果、分项分数、解释特征。
- 独立评估：MRR、NDCG@K、Precision@K、覆盖率、多样性、冷启动鲁棒性。

## 公共接口与数据约定
- `student_profile`：`grade`、`major`、`skills[]`、`interests[]`、`experience_summary`、`availability`
- `mentor_candidate`：`mentor_id`、`topic_score`、`graph_score`、`community_id`、`final_score`
- `project_candidate`：`project_id`、`mentor_id`、`required_skills[]`、`fit_score`
- `teammate_candidate`：`student_id`、`complementarity_score`、`shared_interest_score`
- `final_recommendation`：导师主列表，每项附带推荐原因、关联项目、关联队友、可视化子图
- Skill 间通信原则：只通过共享数据层和结构化输出交互，不依赖彼此内部实现，符合作业要求的模块化模式。

## 可视化与结果呈现
- 必备可视化 1：学生为中心的 ego network，展示学生、目标导师、相关项目、潜在队友之间的连接。
- 必备可视化 2：导师社区图或主题簇图，说明推荐不是黑箱排序。
- 最终结果页结构：学生画像摘要，Top-K 导师，导师对应的项目入口，建议联系/合作的队友，推荐理由摘要。

## 评估与测试
- 主评估方案：混合评估。
- 离线指标：导师推荐使用 `Precision@K`、`Recall@K`、`MRR`、`NDCG@K`；项目/队友使用相关性评分、覆盖率、多样性。
- 人工评估：由小组成员或邀请同学对推荐结果做相关性、可信度、解释清晰度打分。
- 场景测试 1：低年级、无科研经历、只有兴趣标签的冷启动学生。
- 场景测试 2：有课程项目和简历、目标方向明确的中高年级学生。
- 场景测试 3：兴趣跨领域，导师网络稀疏，检验长尾方向推荐能力。
- 场景测试 4：技能与目标项目不完全匹配，检验系统是否能通过“队友补位”给出可执行路径。

## 报告与交付对应
- Group Report 聚焦：Agent 任务定义、5 个 Skill 集成方式、完整推荐链路、整体评估结果、系统可视化。
- Individual Report 聚焦：各自 Skill 的功能、方法、实验与分析。
- StudyClawHub 发布要求：每个 Skill 和 Agent 都需要清晰的名称、描述、版本、标签、作者与 GitHub 地址。
- 汇报演示建议：现场展示一个学生画像输入，跑出导师主推荐，并可点开展示项目与队友解释链路。

## 默认假设
- 数据采用“混合数据”方案：公开学术图谱数据负责导师与研究关系，模拟校园数据负责本科项目和学生画像。
- 首版主任务固定为导师匹配；项目和队友只作为辅助推荐，不单独成为主排序对象。
- 用户输入默认包含基础表单信息和简历/经历文本，不要求完整学术主页或论文列表。
- 第五个 Skill 优先投入到更强的排序算法，而不是单独做对话层；解释与可视化作为 Agent 输出能力统一完成。
- 若真实校园数据不足，可构造半合成 benchmark，以保证结果可测、可复现、可写入课程报告。
