# ProgRec LLM-First Chat Agent and Skill 3 Reasoning Design

> Target: replace the current command-shaped REPL behavior with an LLM-first conversational agent, while upgrading Skill 3 from rule-written mentor reasons to LLM-generated evidence-grounded explanations.

## Goal

Turn `progrec_agent` into a real chat-first ProgRec agent that:

- understands natural-language requests before deciding whether to answer, clarify, refuse, or run repository skills
- behaves as a bounded recommendation-domain agent instead of a keyword-based command wrapper
- answers session meta-questions directly
- clearly rejects out-of-scope questions instead of misrouting them into recommendation flows
- upgrades Skill 3 mentor explanations so the evidence remains structured, but the final explanation text is generated by an LLM from that evidence

## Approved Product Decisions

The design below reflects explicit user choices gathered during brainstorming:

- interaction model: pure chat, not command-first
- startup UX: print a short introduction plus example prompts
- recommendation entry: if context is missing, ask the smallest necessary follow-up question first
- domain boundary: domain-first; answer ProgRec tasks and session meta-questions, but refuse unrelated questions such as weather
- routing strategy: LLM-first, with local logic only for narrow safety and failure handling
- no-LLM behavior: block conversational mode instead of pretending to be intelligent with local heuristics
- Skill 3 explanation output: keep existing structured evidence and `reasons`, and add a new LLM-written `reason_text`

## Problem Statement

The current REPL still behaves like a command system with a free-text escape hatch:

- `repl.py` exposes explicit commands and only forwards unmatched text to the agent layer
- `intent_router.py` falls back to a small keyword router
- unknown or weakly classified messages degrade into a generic clarification prompt
- meta-questions such as "which skill did you use just now?" are not treated as first-class intents
- out-of-scope questions are not clearly bounded

This produces the exact failure mode the user called out: the system feels like a rule model wearing an agent costume.

Skill 3 has a similar issue on the explanation side:

- mentor candidate evidence is already strong and structured
- explanation text is still composed by deterministic template logic in `skill3_mentor_discovery/explanations.py`
- the output is fact-based but not especially natural, adaptive, or persuasive

## User Experience Target

The new REPL should feel like a bounded conversational agent with tool use, not like a hidden command parser.

When the user starts the REPL, the first output should be a short introduction such as:

```text
ProgRec Agent

I help you explore mentor, project, and teammate recommendations based on your academic interests and goals.

You can talk to me naturally. For example:
- Find me an NLP mentor.
- I’m interested in trustworthy AI and I only have 4 hours per week.
- Show me the current profile of the top mentor.
- Why did you recommend this mentor?
- Check whether my graph-mode artifacts are valid.

If your question is outside the recommendation workflow, I’ll tell you clearly instead of guessing.
```

Representative behaviors:

- `Find me an NLP mentor`
  - if no student context exists, ask one necessary question about whether to use an existing `student_id` or build a profile from conversation
- `Which skill did you use in the last response?`
  - answer from session state without rerunning anything
- `What is the weather today?`
  - say clearly that the question is outside ProgRec's scope

## Architecture

### 1. Chat Frontend

`progrec_agent/repl.py` becomes a thin natural-language frontend:

- print startup introduction
- read user text
- pass the turn to the conversation agent
- support a tiny control vocabulary only for exit, such as `exit` and `quit`

All recommendation, explanation, and inspection behavior moves behind the agent layer.

### 2. Conversation Agent Core

`progrec_agent/agent_core.py` becomes the main decision point for every turn:

1. load conversation state
2. classify the new message through the LLM-first router
3. decide whether the turn is `answer_only`, `clarify_then_wait`, `execute_tool`, or `refuse_out_of_scope`
4. call internal tools only when needed
5. synthesize a user-facing reply
6. record what happened for later meta-questions

### 3. Internal Tool Layer

Existing repository-local actions remain tools, but they stop acting like user-visible commands:

- `recommend_full_pipeline`
- `show_recommended_mentor_profile`
- `show_current_profile`
- `inspect_artifacts`
- `debug_graph_mode`
- `rebuild_skill2_graph`
- any future Skill 1 rebuild action

The agent owns the decision to call them. The user only speaks in natural language.

## Conversation State

`AgentSession` should carry enough state to support true conversational behavior.

Required fields:

- `conversation_history`
- `student_profile`
- `resource_context`
- `skill3_result`
- `skill4_result`
- `skill5_result`
- `pending_clarification_questions`
- `pending_goal_text`
- `pending_confirmation_action`
- `last_router_decision`
- `last_response_summary`
- `last_action_kind`
- `last_tool_name`
- `last_tool_arguments`
- `last_action_result_summary`

These fields let the agent answer questions like:

- what did you just do
- which skill did you use
- do we already have recommendation results
- what profile are you currently using

without rerunning the pipeline.

## Routing Model

The router should stop acting like a thin intent classifier and instead return a structured per-turn decision proposal.

Suggested output shape:

```json
{
  "message_type": "domain_task",
  "intent": "recommend_mentor",
  "in_scope": true,
  "needs_clarification": true,
  "clarification_question": "Do you want to continue with an existing student profile, or should I first build a minimal profile from your interests?",
  "answer_only": false,
  "tool_name": "",
  "tool_arguments": {},
  "meta_reply": "",
  "reasoning_summary": "The user wants mentor recommendations but there is not enough student context to run the pipeline safely."
}
```

### Required `message_type` categories

- `domain_task`
- `meta_question`
- `out_of_scope`
- `startup_help`
- `unsafe_or_blocked`

### Required intent coverage

- `recommend_mentor`
- `recommend_project`
- `recommend_teammate`
- `inspect_current_mentor`
- `explain_recommendation`
- `show_current_profile`
- `inspect_artifacts`
- `debug_graph_mode`
- `ask_last_action`
- `ask_capabilities`
- `out_of_scope_other`

### Router rules

The prompt must encode these hard requirements:

- ProgRec is not a general chat bot
- out-of-scope questions must not be mapped into recommendation work
- session meta-questions should be answered directly when possible
- only one clarification question should be asked at a time
- tools should be proposed only when the user intent and required context are both present
- if no recommendation result exists yet, the router must not pretend that a recommendation can already be explained

## LLM Availability Policy

The user explicitly chose not to support a fake-smart fallback mode.

Required behavior:

- if no LLM credentials are available, conversational REPL startup should stop with a clear error and configuration guidance
- if an in-session LLM call fails, the system may use narrow local safeguards for control flow, but it should not quietly degrade into a pretend agent
- fallback logic may still handle only minimal cases such as clean exit, pending confirmation cancellation, and structured failure reporting

This is intentionally stricter than the current local keyword router.

## Response Policy

The response synthesizer should support four reply kinds.

### 1. Direct answer

Use when the user asks a meta-question or requests startup help.

Examples:

- explain last tool or last skill use
- summarize current profile
- describe what the agent can do

### 2. Clarification

Use when the task is in scope but missing the smallest necessary context.

Rules:

- ask only one question
- ask for the next missing decision, not the full profile
- reuse context already present in session state

### 3. Tool-backed reply

Use when the task is in scope and enough context exists to execute an internal tool safely.

### 4. Scope refusal

Use when the request is unrelated to the recommendation workflow.

Recommended style:

`That question is outside ProgRec's recommendation scope. I can still help with mentor, project, teammate, or graph-debug questions.`

## REPL Interaction Changes

The current command-first help text should be removed as the primary onboarding model.

### Keep

- `exit`
- `quit`

### Remove as primary UX

- `recommend`
- `show mentor <id>`
- `show profile`
- `show trace`

Read-only capabilities behind those commands should remain available through natural language.

## Skill 3 Explanation Upgrade

### Current state

`skill3_mentor_discovery/retrieval.py` computes strong evidence fields already:

- `overlap_terms`
- `meta_path_breakdown`
- `top_evidence_paths`
- `graph_confidence`
- `community_id`
- `activity_score`
- other ranking features such as authority and proximity

But `skill3_mentor_discovery/explanations.py` currently turns them into explanation text through fixed templates.

### New design

Skill 3 explanations should be split into two layers:

1. `reason evidence`
   - structured, deterministic, auditable
2. `reason text`
   - LLM-generated, grounded only in provided evidence

### Evidence contract

The explanation generator should receive a compact evidence payload per mentor candidate, including:

- `mentor_id`
- `mentor_name`
- `overlap_terms`
- `meta_path_breakdown`
- `top_evidence_paths`
- `graph_confidence`
- `community_id`
- `activity_score`
- `topic_score`
- `graph_score`
- `personalized_proximity`
- `mentor_authority`

### Required LLM constraints

The LLM-generated explanation must:

- use only the provided evidence
- not invent projects, graph paths, or mentor traits
- mention the strongest one or two evidence families
- explicitly sound more cautious when `graph_confidence` is low
- fall back to topic-fit-first reasoning when graph evidence is weak or absent
- stay concise enough for terminal and JSON output

### Output shape

The user explicitly chose to keep compatibility while adding richer text.

Each `MentorCandidate` should expose:

- existing `reasons`
- new `reason_text`
- existing evidence fields such as `meta_path_breakdown`, `top_evidence_paths`, and `graph_confidence`

This preserves debuggability and downstream compatibility.

### Failure behavior

If LLM explanation generation fails during Skill 3:

- do not drop the candidate
- preserve structured evidence
- preserve or regenerate short deterministic `reasons`
- set `reason_text` to a deterministic fallback explanation derived from the same evidence

This keeps Skill 3 usable for batch runs and tests.

## Implementation Stages

### Stage 1: chat-first REPL shell

- add startup introduction
- remove command-menu-first UX
- block startup when no LLM config exists

### Stage 2: richer routing and policy

- replace the current thin router output with message-type-aware decisions
- support `meta_question` and `out_of_scope`
- eliminate the generic repeated clarification prompt

### Stage 3: stronger session memory

- record last action metadata
- support direct answers about the current conversation state

### Stage 4: Skill 3 LLM reasoning

- extract structured evidence
- add constrained LLM explanation generation
- keep deterministic fallback behavior
- expose `reason_text`

### Stage 5: regression and conversation tests

- add REPL conversation tests
- add Skill 3 explanation tests
- verify that existing inspection and recommendation flows still work

## Acceptance Criteria

The implementation is complete when all of the following are true.

### Conversational agent

- startup prints a natural-language introduction with example prompts
- the user can begin with `Find me an NLP mentor`
- when context is missing, the agent asks one necessary follow-up question
- `Which skill did you use in the last response?` is answered directly from session state
- out-of-scope questions such as weather are refused clearly and are not mapped into recommendation actions
- without LLM configuration, the conversational REPL does not start in pretend-agent mode

### Skill 3 explanations

- every mentor candidate still includes structured evidence fields
- every mentor candidate includes a new `reason_text`
- `reason_text` reflects the provided evidence rather than generic praise
- low-confidence graph evidence produces more cautious wording
- if the explanation LLM fails, deterministic fallback reasons remain available

### Safety and compatibility

- existing recommendation ranking behavior remains intact unless explanation generation is the only changed behavior
- existing consumers of `reasons` do not break
- debug and inspection paths remain read-only unless explicit rebuild confirmation is granted
